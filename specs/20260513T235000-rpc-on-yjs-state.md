## The Finding

Three findings forced a rewrite.

**Finding 1: Awareness is the wrong protocol for our topology.** Peer-gossip CRDT over a hub-and-spoke wire. The AWARENESS_ATTESTED envelope exists only to override awareness's "trust the publisher" default. Two protocols cooperating to undo each other's defaults is a smell.

**Finding 2: Adding a third protocol (HELLO/ROSTER) just moved the smell.** New wire envelopes, new state machine, new hibernation persistence problem (2 KB attachment cap vs 10 KB manifest), new RPC re-addressing. The same shape rebuilt in a different language.

**Finding 3: The Y.Doc is already a reliable, hibernation-safe, sync-protocol-delivered state channel.** It runs on `YKeyValueLww` (`packages/workspace/src/document/y-keyvalue/y-keyvalue-lww.ts`), which gives `O(active data)` storage with tombstones GC'd to ~22 bytes per writer. Every problem the previous spec solved with new wire envelopes is solved cheaper by writing to a reserved Y.Array.

## The Call Type

```ts
import type { Result } from 'wellcrafted/result';

export type Call = {
  to: string;                                       // target connId (per-socket)
  action: string;                                   // snake_case action key
  input: unknown;                                   // action input
  sent_at: number;                                  // caller-side timestamp
  response: Result<unknown, DispatchError> | null;  // null = pending
};
```

Five fields. Every field load-bearing:

- `to`: required, concrete `connId`. **Not** `replicaId`. See "Why connId addressing" below.
- `action`: required. Dispatch key.
- `input`: required. Action payload.
- `sent_at`: read only by the boot-time orphan sweep.
- `response`: `null` is the pending discriminator. Non-null is terminal. Carries `Result<O, E>` directly using wellcrafted's standard shape (`{ data, error }` per `wellcrafted/src/result/result.ts:12`).

Deliberately absent:

- `from`: caller closes over its own row id; target routes by `to` alone.
- `completed_at`: redundant with the LWW entry's `ts` (set when `response` is written).
- `status`: redundant with `response === null`.
- `requesterClientId` / `targetClientId`: the public model never exposes Yjs `Doc.clientID`. Routing is by `connId`.

### Why connId addressing (not replicaId)

A `replicaId` identifies an install. One install can have multiple sockets (multi-tab on the same browser). If `to` were `replicaId`, every tab's observer would see the call, and every tab would race to write the response. Two handler invocations per call, possibly two side effects, LWW deciding which response the caller sees. Not acceptable.

`connId` is per-socket and is unique by construction. Exactly one observer matches `to === connId`, so the handler runs exactly once.

The cost: callers that know `replicaId` but not `connId` must resolve via the presence map. The presence surface deliberately exposes only `list` + `observe` (see "Why no `find` on `peers`" below); call sites do the array operations explicitly:

```ts
const target = collab.peers
  .list()
  .find((p) => p.replicaId === replicaId);   // PresenceEntry | undefined
if (!target) { /* offline */ }
await collab.dispatch('action', input, { to: target.connId, signal });
```

For "any tab on that install" semantics, `list()` is pre-sorted by `connId` ascending, so the first `.find` match is deterministic across callers. For "this specific tab" the caller does `list().find((p) => p.connId === id)`. For fan-out, `list().filter(...).forEach(dispatch)`.

### Why no `find` on `peers`

One-sentence test: `peers` is a read-only view of live connection rows so callers can choose one concrete connection to dispatch to.

A `find({ replicaId })` method fails that test by doing two things at once: it discovers (give me one) and it hides a tie-break policy (which one, when there are several tabs on one install). The call site then reads as "the macbook peer," but the real shape is one replicaId to many connIds. Make the choice explicit at the call site instead:

```ts
// pick any tab on a given install
const target = collab.peers
  .list()
  .find((p) => p.replicaId === replicaId);

// exact connection
const target = collab.peers
  .list()
  .find((p) => p.connId === connId);

// every tab on an install (fan-out)
for (const p of collab.peers.list().filter((p) => p.replicaId === id)) {
  await collab.dispatch('action', input, { to: p.connId, signal });
}
```

The surface stays small: `list` and `observe`. Add `get(connId)` later only if exact connection lookup becomes noisy or perf-relevant.

## Errors

```ts
import { defineErrors, type InferErrors } from 'wellcrafted/error';
import { Ok, Err, type Result } from 'wellcrafted/result';

export const DispatchError = defineErrors({
  Cancelled: ({ reason }: { reason: unknown }) => ({
    message: 'Dispatch was cancelled',
    reason,
  }),
  ActionNotFound: ({ action }: { action: string }) => ({
    message: `Target has no handler for "${action}"`,
    action,
  }),
  ActionFailed: ({ action, cause }: { action: string; cause: unknown }) => ({
    message: `Action "${action}" failed`,
    action,
    cause,
  }),
});
export type DispatchError = InferErrors<typeof DispatchError>;
```

Three variants. `Cancelled.reason` carries `signal.reason` directly; callers distinguish timeout from user-cancel by checking `reason instanceof DOMException && reason.name === 'TimeoutError'`. No separate `Timeout` variant: the dispatcher has no way to recover the original `ms`, and `signal.reason` is the canonical source.

### Caller side


## Implementation

### Dispatch

The `YKeyValueLww` observer is `(changes: Map<string, KvStoreChange<T>>, origin: unknown) => void`. `KvStoreChange<T>` is a discriminated union (`observable-kv-store.ts:28`):

```ts
type KvStoreChange<T> =
  | { action: 'add'; newValue: T }
  | { action: 'update'; newValue: T }
  | { action: 'delete' };
```

`observe()` returns `void`; remove via `unobserve(handler)`. After narrowing on `change.action !== 'delete'`, `change.newValue` is a required `T`.

```ts
import type { KvStoreChange } from '@epicenter/workspace';

async function dispatch<I, O>(
  rpc: YKeyValueLww<Call>,
  action: string,
  input: I,
  { to, signal }: DispatchOptions,
): Promise<Result<O, DispatchError>> {
  const id = generateGuid();
  rpc.set(id, { to, action, input, sent_at: Date.now(), response: null });
  try {
    return await waitFor<O>(rpc, id, signal);
  } finally {
    rpc.delete(id);
  }
}

function waitFor<O>(
  rpc: YKeyValueLww<Call>,
  id: string,
  signal: AbortSignal,
): Promise<Result<O, DispatchError>> {
  const { promise, resolve } =
    Promise.withResolvers<Result<O, DispatchError>>();
  const cleanups: Array<() => void> = [];
  let settled = false;

  const settle = (value: Result<O, DispatchError>) => {
    if (settled) return;
    settled = true;
    for (const fn of cleanups) fn();
    resolve(value);
  };

  // (1) response arrives via YKeyValueLww observer
  const handler = (changes: Map<string, KvStoreChange<Call>>) => {
    const change = changes.get(id);
    if (!change || change.action === 'delete') return;
    const { response } = change.newValue;
    if (response) settle(response as Result<O, DispatchError>);
  };
  rpc.observe(handler);
  cleanups.push(() => rpc.unobserve(handler));

  // synchronous pre-check (response may have already arrived)
  const existing = rpc.get(id);
  if (existing?.response) {
    settle(existing.response as Result<O, DispatchError>);
    return promise;
  }

  // (2) external cancel (covers timeout via AbortSignal.timeout)
  const cancel = () => settle(DispatchError.Cancelled({ reason: signal.reason }));
  if (signal.aborted) {
    cancel();
  } else {
    signal.addEventListener('abort', cancel, { once: true });
    cleanups.push(() => signal.removeEventListener('abort', cancel));
  }

  return promise;
}
```

Properties:

- One `settle()` path. Cleanups run on first settle regardless of winner.
- No internal `AbortController`. No `.error` unwrapping.
- `Cancelled.reason` is whatever the caller's signal aborted with. `AbortSignal.timeout(ms)` aborts with `DOMException('TimeoutError')`; `AbortController.abort('user-cancel')` aborts with `'user-cancel'`. Caller-side narrowing.

### Target observer

```ts
function attachActionRunner(
  rpc: YKeyValueLww<Call>,
  selfConnId: string,
  actions: ActionRegistry,
): () => void {
  const handler = (changes: Map<string, KvStoreChange<Call>>) => {
    for (const [id, change] of changes) {
      if (change.action === 'delete') continue;
      const call = change.newValue;
      if (call.to !== selfConnId || call.response !== null) continue;

      const respond = (response: Result<unknown, DispatchError>) => {
        rpc.set(id, { ...call, response });
      };

      const target = actions[call.action];
      if (!target) {
        respond(DispatchError.ActionNotFound({ action: call.action }));
        continue;
      }

      void run(target, call.input).then((result) =>
        respond(
          result.error
            ? DispatchError.ActionFailed({ action: call.action, cause: result.error })
            : Ok(result.data),
        ),
      );
    }
  };
  rpc.observe(handler);
  return () => rpc.unobserve(handler);
}

async function run(action: Action, input: unknown): Promise<Result<unknown, unknown>> {
  try {
    const ret = action.input !== undefined ? await action(input) : await action();
    return isResult(ret) ? ret : Ok(ret);
  } catch (cause) {
    return Err(cause);
  }
}
```

The read-modify-write preserves `to`, `action`, `input`, `sent_at` via spread; only `response` is added. The LWW `ts` ensures the terminal write wins over the original pending write. `run` is the inlined replacement for `invokeActionForRpc` (`shared/actions.ts:368`): it returns whatever the handler produced, plus `Err(cause)` for thrown errors. The observer wraps any non-null error into `DispatchError.ActionFailed`, so the wire boundary stays `Result<O, DispatchError>` without an intermediate `RpcError` type.

### Client-minted connId

The client mints `connId` (per-`openCollaboration` lifetime, `crypto.randomUUID()`) and passes both ids on the WebSocket URL: `?replicaId=<flat>&connId=<flat>`. Client-minted because the target observer needs to know its own connId synchronously, before the socket opens. Server-minting would require either echoing connId back over a new envelope (rejected) or polling the presence map for self (race-prone).

The server treats connId as opaque routing data. No trust assertion on connId; it's effectively a per-socket session id.

### Server-side presence writer

The server is the only writer to `presence`. It echoes the client-claimed connId and replicaId, stamps the auth-derived subject.



`SERVER_ORIGIN` is a `Symbol`. It distinguishes server presence writes from peer writes in `doc.on('updateV2', ...)` listeners and from server-write-validation (see below).

#### Rejecting peer writes to `presence`

Peers must not write to `PRESENCE_KEY`. `Y.applyUpdateV2` has no "preview affected types" API, so validate by applying to a throwaway doc and inspecting:

```ts
function updateTouchesPresence(payload: Uint8Array): boolean {
  const scratch = new Y.Doc({ gc: true });
  Y.applyUpdateV2(scratch, payload);
  return scratch.getArray(PRESENCE_KEY).length > 0;
}
```

In `applyMessage` (`sync-handlers.ts`), when `messageType === MESSAGE_TYPE.SYNC` and `syncType` is either `STEP2` or `UPDATE` (both paths feed `Y.applyUpdateV2`), call `updateTouchesPresence(payload)`. If true, drop the frame and close 4400. STEP1 is a state-vector probe, no validation needed. The scratch-doc decode cost is O(update bytes), bounded by `MAX_PAYLOAD_BYTES`, on the same path that's already decoding the update.

Note: `RPC_KEY` is peer-writable. Both clients write to it (caller posts, target responds). No validation there.

## Wire Protocol Changes

```
DELETE (envelope + codecs):
  MESSAGE_TYPE.AWARENESS              (1)
  MESSAGE_TYPE.QUERY_AWARENESS        (3)
  MESSAGE_TYPE.AWARENESS_ATTESTED     (100)
  MESSAGE_TYPE.RPC                    (101) + RPC_TYPE.{ACTION_REQUEST, RESPONSE, RUNTIME_REQUEST}
  encodeAwareness, encodeAwarenessStates, encodeAwarenessAttested
  encodeQueryAwareness, decodeAwarenessAttestedPayload
  encodeRpcActionRequest, encodeRpcRuntimeRequest, encodeRpcResponse
  decodeRpcMessage, decodeRpcPayload
  RuntimeVerb, RPC_TYPE, DecodedRpcMessage

KEEP:
  MESSAGE_TYPE.SYNC                   (0)   STEP1 / STEP2 / UPDATE
  MESSAGE_TYPE.AUTH                   (2)   4401 close path only; no traffic
```

After deletion, the only `MESSAGE_TYPE` values that produce wire bytes are `SYNC` (0). `AUTH` (2) is a reserved sentinel for the 4401 permanent-close code; no frames are ever exchanged. The protocol becomes byte-identical to plain y-websocket sync.

## Honest Downsides

1. **Server is an explicit Yjs writer to a reserved root.** It was already a Yjs participant (`Y.applyUpdateV2` on HTTP sync at `base-sync-room.ts:277`, compaction rewrites the log at `base-sync-room.ts:556`), but this is the first time it writes a *named* shared type. Mitigated by the reserved `PRESENCE_KEY` and the scratch-doc validation that drops peer updates touching that root.

2. **Capability discovery is app-level.** Apps that want "list peers offering action X" build their own table. The only consumer with multi-host needs today is `tab-manager`, and its model is already device-table-shaped (`apps/tab-manager/src/lib/device.ts:50`).

3. **In-flight calls leak storage briefly on caller crash.** If the caller crashes between writing the request and reaching `finally { rpc.delete(id) }`, the entry persists. The next `openCollaboration` on this workspace runs a client-side sweep before attaching the action runner:
   ```ts
   const cutoff = Date.now() - 1000 * 60 * 60; // 1h
   for (const [id, entry] of rpc.entries()) {
     if (entry.val.sent_at < cutoff) rpc.delete(id);
   }
   ```
   Server-side: not needed. The server never reads rpc rows; it just persists the bytes. The next workspace open eventually deletes them.

4. **Tombstones grow on `gc: false` docs.** Workspace docs are `gc: true` (`base-sync-room.ts:146`). Document docs are `gc: false` to preserve snapshot history. RPC and presence MUST live only on workspace docs; openCollaboration enforces this by being the only API that constructs the YKeyValueLww at these keys.

5. **No `to: '*'` (any-host) dispatch.** Caller must address a specific `connId`. Most callers know the replicaId and resolve via `peers.list().find((p) => p.replicaId === id)?.connId`; multi-host fan-out is `peers.list().filter(...).forEach(dispatch)`. The dispatcher stays single-target, and the surface refuses to bake a tie-break choice into a helper.

6. **One scratch-doc apply per inbound STEP2/UPDATE.** The presence write-check applies the incoming bytes to a throwaway `Y.Doc` to inspect `getArray(PRESENCE_KEY).length`. Cost is bounded by `MAX_PAYLOAD_BYTES`. RPC traffic produces small deltas; bulk-write paths (large UPDATE frames) pay proportionally. Yjs has no rewind primitive, so "accept then revert" is not an alternative; the scratch-doc check is the cheap path.

## Resolved Decisions

1. **Identity handshake = query param.** `?replicaId=<flat>&connId=<flat>` on the WebSocket URL. Both client-minted, server-echoed; subject is server-stamped. Zero new envelopes.

2. **`createReplicaIdAsync` stays.** Tab-manager's `chrome.storage` is the consuming use case (`apps/tab-manager/src/lib/device.ts:50`). 12 lines, one real consumer; removing it would force tab-manager into a custom helper.

3. **`presence` write validation = scratch-doc apply.** Apply the inbound update bytes to a throwaway `Y.Doc`, reject if `scratch.getArray(PRESENCE_KEY).length > 0`. One extra `Y.applyUpdateV2` per inbound STEP2 or UPDATE frame (and per HTTP `sync()` body). The alternative "accept then roll back" path doesn't exist; Yjs has no rollback primitive. Cost is bounded by `MAX_PAYLOAD_BYTES`.

4. **`gc: true` invariant = enforced by API surface.** `openCollaboration` is the only callsite that constructs the RPC and presence stores. Document docs use `attachYjsSync` (no actions, no presence). Cross-coupling is structurally impossible.

## Deferred

- **Server-origin Yjs clientID.** Each DO's `Y.Doc` instance gets a random `clientID` at construction (`base-sync-room.ts:146`). Collisions with peer clientIDs are theoretically possible but irrelevant to LWW conflict resolution (the comparator is `ts`, with positional tiebreak; `clientID` is not consulted by `YKeyValueLww`). Revisit only if a future feature reads `clientID` from presence rows.

## What This Does Not Touch

```
Y.Doc sync protocol (SYNC_STEP1/2/UPDATE)   unchanged
attachIndexedDb / BroadcastChannel          orthogonal
attachTable / attachKv / attachTimeline     orthogonal
@epicenter/encryption                       orthogonal
ActionRegistry / defineQuery / defineMutation  unchanged
```

The blast radius is identity + presence + RPC. Resist bundling adjacent cleanups.

## Review

**Completed**: 2026-05-13
**Branch**: `codex/sync-room-plus-stacked-refactors`

### Summary

The RPC plane and identity plane were both collapsed onto Y.Doc state. The `@epicenter/sync` package now exports only Yjs sync codecs and auth-subprotocol helpers; `AWARENESS`, `AWARENESS_ATTESTED`, `QUERY_AWARENESS`, and `RPC` envelope kinds (plus their codecs, `RpcError`, and `RuntimeVerb`) are gone. The workspace ydoc now reserves two top-level Y.Arrays: `RPC_KEY = 'rpc'` carries `YKeyValueLww<Call>` rows that the caller writes and the target flips from `response: null` to a `Result`, and `PRESENCE_KEY = 'presence'` carries server-written `PresenceEntry` rows. `openCollaboration` mints a per-socket `connId`, wraps `openWebSocket` to append `?replicaId=&connId=` on every connect, and exposes a `dispatch(action, input, { to, signal })` method plus a read-only `peers` surface. The DO base class writes presence rows in `upgrade()`, runs a boot orphan sweep against the live `ctx.getWebSockets()` set, and validates inbound STEP2/UPDATE/HTTP payloads against a scratch-doc check that rejects any write to `PRESENCE_KEY` with a 4400 close (or HTTP 403 on the `sync()` path).

### Deviations from Spec

- **`peers.find` removed.** The spec described `peers.find({ connId? } | { replicaId? })` returning a `PresenceEntry`. During implementation the one-sentence test ("peers is a read-only view of live connection rows so callers can choose one concrete connection to dispatch to") flagged that `find({ replicaId })` was secretly doing tie-break policy. The surface was cut to `{ list, observe }`; callers do `peers.list().find((p) => p.replicaId === id)` explicitly. The spec body was updated in-place to document this and added a "Why no `find` on `peers`" section.
- **`invokeAction` signature simplified.** Wave 4's agent dropped the third `errorLabel` argument and let `invokeAction` return `Result<T, unknown>` directly (raw thrown cause under `.error`). The wire boundary in `attachActionRunner` now wraps any non-null error into `DispatchError.ActionFailed`, so call sites no longer need to thread an error label.
- **`computeInitialMessages` deleted.** With awareness gone, initial messages reduced to a single `encodeSyncStep1` frame, which is now inlined at the upgrade callsite in `base-sync-room.ts`. The exported helper had no remaining callers.
- **`PresenceWriteForbidden` HTTP 403 mapping added.** Wave 5 threw `PresenceWriteForbidden` from `BaseSyncRoom.sync()`; Wave 6 added try/catch wrappers at both `stub.sync(body)` callsites in `apps/api/src/app.ts` mapping that error to `c.body('presence-write-forbidden', 403)`.
