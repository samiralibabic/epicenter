# Logger Vision, What `wellcrafted/logger` Is, And Isn't

**Date**: 2026-05-15
**Status**: decided
**Scope**: `~/Code/wellcrafted/src/logger` and its consumers in `~/Code/epicenter`

## One-sentence purpose

Bind a `source` string to a sink, where the sink is usually a `composeSinks` fan-out across `consoleSink` plus an app-built destination (jsonl file at the daemon, toast/notification interception at the UI), then call `log.warn(typedError)` / `log.error(typedError)` / `log.info(msg, data?)` from library code, and tests swap the whole sink for `memorySink` to assert on captured events.

Console-only is the degenerate case of that sentence; fan-out is the typical case.

## What it is

A minimal binding layer between sync caller code (`log.warn(err)`) and pluggable destinations (`(event) => void`). Four primitives, one combinator, two reference sinks:

```
Primitives
├── createLogger(source, sink?)     bundles 5 level methods over a sink
├── LogEvent                         normalized event shape
├── LogSink                          (event) => void & Partial<AsyncDisposable>
└── Logger                           5 methods: trace/debug/info/warn/error

Reference sinks
├── consoleSink                      dev default; console[level] with [source] prefix
└── memorySink()                     { sink, events }; the load-bearing test affordance

Combinator
└── composeSinks(...sinks)           fan-out; forwards dispose

Result-flow helper
└── tapErr(logFn)                    .inspect_err for Result pipelines
```

That is the entire surface. Roughly 200 lines total, ~30 lines of which carry the actual semantics.

## What it isn't

- **Not a tracing system.** No spans, no ambient context, no `withContext`. Scope propagation is OpenTelemetry territory; build it when you need it, separately.
- **Not a durability layer.** No file sinks, no rotation, no buffering policy, no backpressure machinery. The host (shell, systemd, Cloudflare tail) owns durable storage; the logger emits to whatever the consumer points it at.
- **Not a global registry.** No `setDefaultLogger`, no module-level state. Each consumer takes `log?: Logger` and defaults to a console-bound logger for dev ergonomics.
- **Not a format authority.** Sinks decide how events are rendered. The library does not ship "the JSONL format" or "the pretty format."
- **Not a level-filter engine.** Today every sink receives every event. If level filtering becomes a real need, it lives in a sink wrapper, not in the core.

## Why this shape (and not just `console.*`)

Honest cost/benefit of the abstraction vs raw `console`:

| What Logger gives you | Worth keeping? |
|---|---|
| `memorySink` for test assertions on structured events | **Yes, structurally.** No epicenter test uses it today (no test asserts on log events yet); kept because the moment one does, console spies lose the typed-error payload. |
| Typed-error binding (`warn(err)`, `error(err)`) where the variant owns the message | **Yes.** 28 call sites today pass a `defineErrors` factory result or a narrowed `result.error`. Pairs with `tapErr` for Result pipelines. |
| `source` as a built-in field instead of a hand-rolled prefix | Marginal, `console.warn('[src]', ...)` does the same. Library makes the convention uniform. |
| Sink composition (`composeSinks`) for fan-out | **Yes, central.** The two most common app wirings are console+jsonl-file (daemon-side, for persistence beyond an attached terminal) and console+UI-toast-interception (app-side, so user-visible warnings render in the product, not just the devtools). Both compose `consoleSink` with an app-built sink; both rely on `Partial<AsyncDisposable>` for clean shutdown of the second destination. |
| Pluggable destinations at all | **Conditional.** Same as above. The *capability* matters for telemetry shipping later; the *implementation* of any non-console sink belongs at the consumer. |

The marginal cost of the abstraction (one import, a vocabulary tax, an injection decision at each factory) is paid even when only the typed-error binding is doing work in-tree. That's acceptable today; if the typed-error binding stops earning its keep, collapse the whole thing.

## The "log to console AND save to JSONL" question

This is the use case that pulled `jsonlFileSink` back into the conversation. Two honest paths:

### Path A, Host-owned durability (the current bet)

The library emits to `consoleSink`. The operator chooses where stdout/stderr go:

```bash
bun run start                              # dev
bun run start 2>> ~/.app/app.jsonl         # ad-hoc file capture
systemd-run --user bun run start           # journal, queryable via journalctl
bun run start | tee ~/.app/app.jsonl       # human view + file copy
```

Pros: zero library code; durability is a problem the OS solved decades ago; no policy decisions (rotation, backpressure, error fallback) inside the library; every standard ops tool just works.

Cons: stdout is unstructured human prose unless you swap `consoleSink` for a structured-stdout sink (~10 lines of JSON.stringify). Sites that want both human-readable dev and JSON-machine prod have to switch sinks at startup, not fan out.

### Path B, In-process fan-out via `composeSinks`

If "human + structured-on-disk in the same process" is a real, recurring requirement, the design supports it natively:

```ts
const sink = composeSinks(consoleSink, myJsonlSink);
const log = createLogger('workspace/sync', sink);
```

`myJsonlSink` lives at the consumer, not in `wellcrafted/logger`. Built when needed, scoped to that consumer's exact requirements (rotation? backpressure? error policy?), with a real caller to constrain the design.

### How to choose

- If you want a file alongside human output **at the operator level**, Path A wins. Shell redirect with `tee`, or a structured-stdout sink and `jq`.
- If you want a file **inside the process** (because the operator can't be relied on, or you need rotation/structure/correlation the shell can't give you), Path B is exactly what `composeSinks` is for. Build the sink then; do not pre-ship it.

The mistake we made and reversed: shipping a `jsonlFileSink` before a caller existed to constrain it. The complexity (backpressure, `onError`, dispose-await) accumulated to satisfy a hypothetical use case. Without a caller, no signal told us which knobs were actually needed.

## Final shape, `~/Code/wellcrafted/src/logger`

```
src/logger/
├── index.ts              public exports only
├── types.ts              LogLevel, LogEvent, LogSink, Logger, LoggableError
├── create-logger.ts      createLogger(source, sink?)
├── console-sink.ts       consoleSink (dev default)
├── memory-sink.ts        memorySink() (test affordance)
└── compose-sinks.ts      composeSinks(...sinks)
```

`tapErr` lives next to `Result` (already correct).

**No file sink. No network sink. No telemetry sink.** Consumers add those when they need them.

## How it's used in `~/Code/epicenter`

### Library code (`packages/workspace`, `packages/auth`)

```ts
// Multi-instance modules (attach primitives, factory builders):
export function attachThing(
  ydoc: Y.Doc,
  { log = createLogger('workspace/thing') }: { log?: Logger } = {},
) { /* ... */ }

// Process-singleton modules (CLI commands, app bootstrap, leaf utilities):
const log = createLogger('workspace/daemon/metadata');
```

Rule: **multi-instance, inject; process-singleton, module-scope.** Test for "multi-instance": could two concurrent instances in the same process want different sinks? If yes, inject.

### App wiring (`apps/<app>/blocks/`)

Single sink, multiple loggers, usually all `consoleSink`:

```ts
attachThing(ydoc, { log: createLogger('workspace/thing') });
attachOther(ydoc, { log: createLogger('workspace/other') });
```

Or composed, when the app has a real second destination:

```ts
const sink = composeSinks(consoleSink, telemetrySink);
attachThing(ydoc, { log: createLogger('workspace/thing', sink) });
attachOther(ydoc, { log: createLogger('workspace/other', sink) });
```

### Tests

Always `memorySink`. Never spy on `console`.

```ts
const { sink, events } = memorySink();
const log = createLogger('test', sink);
await doThing({ log });
expect(events[0]).toMatchObject({ level: 'warn', source: 'test' });
```

### Source convention

`<package>/<module>` or `<app>/<feature>`, lowercase-kebab:

```
'workspace/sync-supervisor'
'workspace/attach-yjs-log'
'auth/oauth-app'
'auth/state-store'
'fuji/daemon-route'
```

Not enforced. Documented in the `createLogger` JSDoc and the `logging` skill. Migrate opportunistically; don't churn for the convention alone. Drift inventory at the bottom of this spec.

## Non-goals (explicit)

1. **No child loggers / `withContext`.** Tried; killed because `unknown` was a tell that the abstraction wasn't load-bearing. If real cross-async-scope correlation becomes a need, solve with `AsyncLocalStorage` + a thin app-level wrapper, not by adding a `context` field to `LogEvent`.
2. **No `bestEffortSink` / sink-error swallowing primitive.** Per-sink concern, per-app policy. The library's job is to ship sinks that handle their own I/O honestly; the consumer composes for fault tolerance if needed.
3. **No level filtering in core.** If `LOG_LEVEL=warn` becomes a need, ship `levelFilterSink(min, inner)` as a wrapping sink, keeps the contract closed, adds a primitive without changing core types.
4. **No format opinionation.** No "the JSONL format," no "the pretty format," no "the OTel format." Sinks decide formatting; the library normalizes only the event shape.
5. **No async `emit`.** `(event) => void` stays. Sinks that own async resources buffer internally and flush on `[Symbol.asyncDispose]`. Pretending emit is async forces every business code path to think about logging concurrency, which is the opposite of what a logger is for.

## Per-primitive decisions

For each primitive: (1) the strongest critique against keeping it as-is, (2) the strongest defense, (3) keep/change/defer with a one-line justification, (4) the extension seam.

---

### 1. `createLogger(source: string, sink: LogSink = consoleSink)` signature

**Critique (delete it):** A `{ source, sink }` object would extend cleanly to a third option (level threshold, scope name, fields) without rewriting every call site. Positional + default puts the source-string convention in JSDoc, where it can drift, instead of in the type. And `source: string` is a free-form text field where `{ pkg, mod }` would let consumers filter on package without parsing.

**Defense (keep it):** Two arguments, both obvious from the name. Twenty-two consumer call sites today; adding `{ source: 'x' }` wrappers at every one pays ceremony for a hypothetical third field that nobody has asked for. `source: string` matches Rust `tracing::target`; structured sources optimize for a filter that doesn't exist in this tree. Sink composition (level threshold, source filter) attaches at the sink layer, not the factory.

**Decision:** KEEP. Two args, both load-bearing today; the third option lives at the sink, not the factory.

**Extension seam:** Future level filtering, source filtering, and format choice all attach as **sink wrappers**, not factory options. `levelFilterSink('warn', consoleSink)` and `sourceFilterSink('workspace/', consoleSink)` wrap the destination; `createLogger` doesn't change. Concrete second consumer for a level-filter seam: the workspace sync supervisor emits four `log.info` lines per status change (`'sync connected'`, `'sync failed'`, `'sync offline'`, `'sync supervisor exited'`) that a noise-conscious operator would want to suppress without losing warn/error.

---

### 2. `LogSink` callable-vs-object shape

**Critique (delete it):** `interface LogSink { emit(event): void; dispose?(): Promise<void> }` would be obvious in TS error messages. The current intersection of a callable with `Partial<AsyncDisposable>` is exotic; the `satisfies LogSink` dance on every sink is a tell that the type fights its users. A future "sink-as-object" extension (named sinks, sinks with config inspection) doesn't fit a callable.

**Defense (keep it):** Pure-function sinks are the common case (`consoleSink` and `memorySink`'s inner sink are both literal arrow functions; an object form would force `{ emit: (e) => ... }` boilerplate at every site), but the `Partial<AsyncDisposable>` half is load-bearing for the two named near-term app sinks: `jsonlFileSink` must flush a buffered write stream on shutdown, and `toastSink` may want to unsubscribe from its UI store. `await using` semantics already require `[Symbol.asyncDispose]` regardless of object-vs-callable; the language chose this surface. `Partial` keeps pure sinks unencumbered while stateful sinks attach dispose via `Object.assign(emit, { [Symbol.asyncDispose]: dispose })`, exactly the pattern `composeSinks` already uses internally to forward dispose across the fan-out.

**Decision:** KEEP. Callable preserves "sink IS a function"; the optional asyncDispose is the named seam for `jsonlFileSink` and `toastSink`, both of which are planned for the daemon and UI consumers respectively.

**Extension seam:** Stateful sinks plug in via `Object.assign(emit, { [Symbol.asyncDispose]: dispose })`. The proof that this seam holds: an earlier `jsonlFileSink` implementation (removed before this spec because it shipped without a caller, not because the shape was wrong) used exactly this composition with `consoleSink` through `composeSinks` and validated the shape end-to-end. When `jsonlFileSink` returns at the daemon consumer, it reuses the same pattern without any core type change.

---

### 3. `LoggableError = AnyTaggedError | Err<AnyTaggedError>` dual shape

**Critique (delete it):** Pick one. Always pass the unwrapped tagged. `log.warn(MyError.X({...}).error)` at every fresh-factory site is three keystrokes, and the union forces a structural discriminator (`"name" in err`) that has already needed a docs paragraph explaining why we don't check `data === null` (Ok-null / Err-null shape collision). One shape, one failure mode.

**Defense (keep it):** Three concrete call patterns appear in the tree, all paying their keep:

```ts
// fresh factory (Err<AnyTaggedError>)
log.warn(EncryptedKvError.DecryptFailed({ key, reason }));

// narrowed result.error (AnyTaggedError)
log.warn(walResult.error);

// native Error (AnyTaggedError, structurally)
log.warn(new Error('db.close() failed during destroy', { cause }));
```

Forcing always-unwrap makes the fresh-factory case ugly at 14 call sites. Forcing always-wrapped requires `Err(result.error)` ceremony at every narrowed-error site. The discriminator is invariant-safe: `defineErrors` stamps `name` on every tagged variant as a hard invariant; `Err` wraps the tagged value as `{ error, data: null }` with no top-level `name`. The structural check is honest about what discriminates, not lucky.

**Decision:** KEEP. Three distinct in-tree call patterns prove the union earns its complexity.

**Extension seam:** A future change to `defineErrors`'s factory return type would flip the discriminator; the change lives in `unwrapLoggable` (`"name" in err`) and the JSDoc. Adding *more* accepted shapes (e.g. a `ResultLike` third arm) extends as an extra union member and one extra branch in `unwrapLoggable`. Trigger to collapse the union: if `defineErrors` ever stops returning `Err<T>` from factory calls (e.g. switches to returning the tagged directly), the `Err`-wrapped arm becomes dead and the type narrows to `AnyTaggedError` alone.

---

### 4. `consoleSink` as default

**Critique (delete it):** Defaults hide where logs go. Requiring explicit injection makes sink wiring visible at every factory boundary, the honest cost. In production, a forgotten injection means console spam that should have been routed.

**Defense (keep it):** Twenty-two multi-instance factories today default to `createLogger('source')`. Removing the default forces every test, every app, every script to pass `log: createLogger('x', consoleSink)`, twenty-two identical lines for zero new signal. Test isolation already lives in `memorySink` (and consumers pass it explicitly when they want it). Production routing already lives in app bootstrap. The default optimizes the 95% dev case; the 5% production case overrides at the factory call site.

**Decision:** KEEP. Default-to-console is the dev-ergonomics win; production overrides at the wiring boundary.

**Extension seam:** Apps that want a different default install it at the *factory* call site:
```ts
const sink = composeSinks(consoleSink, telemetrySink);
attachThing(ydoc, { log: createLogger('workspace/thing', sink) });
```
The library default never changes; the consumer overrides per factory. Concrete second consumer when a structured-stdout variant ships: `apps/fuji/blocks/daemon-route.ts` currently relies on the consoleSink default; switching to `jsonConsoleSink` for `bun run start | jq` happens at fuji's `defineFujiDaemon` boot, not in the library.

---

### 5. `memorySink()` return shape `{ sink, events }`

**Critique (delete it):** Every caller does `const { sink, events } = memorySink()`. The destructuring is ceremony. Fuse them: `const sink = memorySink(); sink({...}); sink.events`. Or, drop the helper entirely and let tests build their own one-line capture array. Today zero epicenter tests use it.

**Defense (keep it):** Two roles, two names. `sink` goes to `createLogger`; `events` goes to assertions. Spreading them at the call site makes the two consumers obvious:
```ts
const { sink, events } = memorySink();
const log = createLogger('test', sink);
expect(events[0]).toMatchObject({ level: 'warn', source: 'test' });
```
Fusing them couples the test's two distinct intents (wire the sink, inspect the buffer). Console spies, the alternative, lose the typed-error payload that the warn-unary signature is specifically built to capture.

**Decision:** KEEP. Two-field return matches the two distinct test concerns; the structural shape survives the spread and reads naturally.

**Caveat:** Today no epicenter test uses `memorySink` (the codebase has no test asserting on log events yet). The primitive is positioned as "load-bearing" because it's the right shape *when* such a test appears. Concrete trigger to migrate: any new test that needs to assert a typed-error payload reached the log, that's the moment to wire `memorySink` instead of letting the log fall to `consoleSink` during the test.

**Extension seam:** A future ring-buffer overload (`memorySink({ capacity: 1000 })`) for long-running stress tests attaches as an options arg without changing the return shape. No real caller today; defer until a test actually accumulates unbounded events.

---

### 6. `composeSinks` throw-propagation policy

**Critique (delete it):** Zero non-test callers anywhere in `~/Code/epicenter` *today*. Apps that fan out could do `[a, b].forEach((s) => s(event))` inline (one line) and `await Promise.all(sinks.map((s) => s[Symbol.asyncDispose]?.()))` on shutdown (one line). The combinator is shipping a contract that no in-tree caller has yet constrained.

**Defense (keep it):** Fan-out is the *centerpiece* of the design, not the edge case. Two concrete near-term app wirings need it:

```ts
// daemon-side: persist logs to disk beyond the operator's terminal
const sink = composeSinks(consoleSink, jsonlFileSink('/path/to/app.jsonl'));
const log = createLogger('fuji/daemon-route', sink);

// app-side: intercept warnings for the UI toast layer
const sink = composeSinks(consoleSink, toastSink(toastStore));
const log = createLogger('whispering/transcription', sink);
```

`jsonlFileSink` and `toastSink` are app-built (per the "no file/network/telemetry sinks shipped from core" rule), but the composition pattern is identical across both, and the `Partial<AsyncDisposable>` half of `LogSink` is load-bearing exactly because `jsonlFileSink` must flush its write stream on shutdown. Inlining a two-line forEach per fan-out site would duplicate the dispose-forwarding logic at every app boundary the moment shutdown correctness matters. The throw-propagation policy (first throwing sink stops fan-out) is the right default: silently swallowing a sink throw hides a real bug; best-effort delivery is a per-sink wrapper concern.

**Decision:** KEEP. Two named near-term app wirings (jsonl-persist at daemons, toast-interception at UI apps) make fan-out the canonical pattern, and the dispose-forwarding inside `composeSinks` is non-trivial enough to deserve one home.

**Extension seam:** A future best-effort wrapper is a per-sink concern: `bestEffortSink(innerSink, onError)` returns a `LogSink` that catches throws and forwards them to `onError`. It composes cleanly with the unchanged `composeSinks`:
```ts
const sink = composeSinks(consoleSink, bestEffortSink(toastSink(toastStore), reportToSentry));
```
No core change required; the seam is "wrap the misbehaving sink, then compose". Concrete second consumer when one ships: a toast sink that throws on store-write errors during teardown shouldn't deafen the console half of the chain.

---

### 7. `Logger` 5 methods (trace/debug/info/warn/error)

**Critique (delete it):** `trace` is unused in epicenter (zero call sites). Cut to four. Also, the asymmetric signatures (`warn(err)` vs `info(msg, data)`) surprise readers, pick one shape and either type both sides or free-form both sides.

**Defense (keep it):** Five levels mirror Rust `tracing` exactly; the cost of including `trace` is three lines of method code and one line of `LogLevel` union member, near-zero. Future deep instrumentation (CRDT update tracing, sync handshake byte-level, dispatch-route tracing) is the natural home for `trace` and shouldn't have to add a level to the vocabulary. The signature split is intentional and load-bearing: `warn`/`error` are *error-binding* methods (the typed variant owns the message; level is a call-site decision on top of the variant); `trace`/`debug`/`info` are *free-form* diagnostics (the level IS the discriminator, no typed variant to carry). Conflating them would force `data: unknown` on warn-message-strings, exactly the path that killed `withContext`.

**Decision:** KEEP. Five levels defer a future trace need at near-zero cost; the signature split protects the typed-error binding.

**Extension seam:** Future structured-trace integration (OpenTelemetry export of `LogEvent`, structured-debug viewer) attaches at the sink layer: an `otelSink` consumes `LogEvent` and emits spans; a `debugInspectorSink` buffers for a UI. The level vocabulary is fixed at 5; sink consumers translate. Concrete second consumer needed before shipping any of these: today's tree has none, defer the seam itself, but the path is sink-layer extension, not method addition.

---

### 8. Warn-unary-typed-error decision (`warn(err)` only, no `(message, data?)` overload)

**Critique (delete it as a rule):** Rust `tracing::warn!("msg")` allows the string form; some short-lived warning sites are ceremony to wrap in a single-use `defineErrors` variant. `log.warn('legacy config field detected', { field })` is fine as one-shot. The current rule forces ceremony.

**Defense (keep the rule):** Every warn site in `~/Code/epicenter` today carries structured fields that benefit from a named variant. Two warn sites in `attach-yjs-log.ts` use `new Error('msg', { cause })` (covered by the dual `LoggableError` shape) when there's no real variant to define; the typed-error path is preserved. Adding an ad-hoc string overload reopens the `data: unknown` footgun, the exact same surface that killed `withContext`. The rule is also a forcing function: when a warn site appears, the developer asks "what is this failure, what does it deserve to be named?" The answer is almost always a `defineErrors` variant the codebase wanted anyway.

**Decision:** DEFER overload addition. Concrete trigger to revisit: when ≥3 distinct call sites would each define a single-use `defineErrors` variant solely to satisfy `warn` (i.e. the variant has no other consumer and exists only to dress a one-line warn-string), add a string-form overload. Today: zero such sites.

**Extension seam:** When the trigger fires, the change is purely additive: `Logger.warn` gains an overload `(message: string, data?: unknown) => void` alongside the existing `(err: LoggableError) => void`. Discriminator at runtime: `typeof first === 'string'`. Zero existing call sites break; the unary-error form remains valid for the 28 sites that already use it. The seam is overload addition in `Logger` + a typeof branch in the `emit` path inside `createLogger`.

---

### 9. Source-string convention (`<package>/<module>` lowercase-kebab)

**Critique (delete the convention):** Documented but not enforced; therefore drifts. Today's drift inventory proves it:

```
slash-form (matches convention):
  workspace/attach-yjs-log, workspace/daemon/metadata, auth/oauth-app, auth/state-store, sync

camelCase factory names:
  attachEncryptedIndexedDb, attachSqliteMaterializer, onLocalUpdate,
  createDisposableCache, loadDaemonConfig

bare kebab (no package prefix):
  machine-auth, machine-tokens-store, markdown-materializer,
  encrypted-kv, standard-schema, fuji-sqlite
```

A free-form `source` string lets each module choose; documenting a non-enforced rule guarantees lint noise. Either enforce it or delete the JSDoc paragraph.

**Defense (keep the convention):** Convergence on `<package>/<module>` makes tail-log grep trivial: `grep '\[workspace/' app.log` filters the whole package; `grep '\[auth/' app.log` filters the auth package. The drift inventory is technical debt accumulated across 17 modules, not evidence the rule is wrong. Migration is opportunistic and one-line per module. The convention is documented in the `createLogger` JSDoc and the `logging` skill.

**Decision:** KEEP convention; migrate opportunistically. Concrete trigger to enforce: when log volume in any single app grows past the point where ad-hoc grep stops working (rule of thumb: ≥10 active source strings printed in a single dev run that an operator wants to slice by package), add an ESLint rule that asserts the `^<package>/[a-z0-9-]+(\/[a-z0-9-]+)?$` regex. Until then, the JSDoc paragraph + migration list below carries the weight.

**Drift migration list** (apply opportunistically; do not refactor as part of this spec):

| current | should be |
|---|---|
| `attachEncryptedIndexedDb` | `workspace/attach-encrypted-indexed-db` |
| `attachSqliteMaterializer` | `workspace/attach-sqlite-materializer` |

**Extension seam:** Source filtering attaches at the sink layer: `sourceFilterSink('workspace/', innerSink)` matches any event whose `source` starts with the prefix. Concrete second consumer when one ships: a dev-only console that wants only `auth/*` logs uses `createLogger(src, sourceFilterSink('auth/', consoleSink))`. The factory doesn't change; filtering lives where filtering belongs.

## Open questions (resolved)

1. **Does the `LoggableError` dual shape earn its keep?** Yes, see primitive #3. Three concrete call patterns prove it. Trigger to collapse: a change to `defineErrors`'s factory return type that retires the `Err<T>` wrapper would make the `Err`-wrapped arm dead.
2. **Should `consoleSink` ship a JSON-structured variant?** Defer. Trigger to ship: when a daemon consumer needs `bun run daemon | jq` over the stdout stream AND `consoleSink`'s `[source] message data` prefix prose breaks that pipeline. Today no such consumer; `fuji-daemon` uses default `consoleSink` and operator-side capture is sufficient.
3. **Should `Logger.warn` get an ad-hoc `(message, data?)` overload?** Defer, see primitive #8. Trigger: ≥3 distinct call sites that would each define a single-use `defineErrors` variant solely to satisfy `warn`. Today zero such sites.

## Acceptance, what "right shape" looks like

- [x] `wellcrafted/logger` exports exactly: `createLogger`, `consoleSink`, `memorySink`, `composeSinks`, `tapErr`, plus the five types (`LogLevel`, `LogEvent`, `LogSink`, `Logger`, `LoggableError`).
- [x] No file/network/telemetry sinks shipped from the core.
- [x] Every multi-instance library factory in epicenter accepts `log?: Logger` (verified at: `attachEncryptedIndexedDb`, `attachYjsLog`, `attachMarkdownMaterializer`, `attachSqliteMaterializer`, `createDisposableCache`, `createEncryptedYkvLww`, `onLocalUpdate`, `createOAuthAppAuth`, `createAuthStateStore`, `loadMachineTokens`, `loginWithOob`, `status`, `logout`, `createMachineAuthClient`, `createSyncSupervisor`).
- [x] Every process-singleton library module uses module-scope `createLogger('<source>')` (verified at: `packages/workspace/src/daemon/metadata.ts`, `packages/workspace/src/shared/standard-schema.ts`, `packages/cli/src/load-config.ts`).
- [x] Tests would use `memorySink`, not `console` spies, if any test asserted on log events. Today no such test exists in epicenter; no `console` spies on logger output exist either. Acceptance is vacuously held; the moment a test needs to inspect log events, the path is `memorySink`.
- [ ] Source strings across the codebase converge on `<package>/<module>` (drift migration list in primitive #9; opportunistic).
- [ ] The `logging` skill matches this spec (not audited as part of this convergence; trigger to revisit: any time the skill is opened for an unrelated edit, sync the four primitives sections inline).

## What would prove this design wrong

- A real, recurring need for cross-async-scope correlation that `AsyncLocalStorage` + app helpers can't cleanly solve.
- A second consumer-defined sink that ends up duplicating significant logic with a hypothetical third, i.e. the consumer-builds-its-own rule producing real code duplication across consumers.
- Operators consistently asking for in-process durability that shell redirect can't satisfy (rotation by size, structured query before write, etc.).
- The two named near-term fan-out wirings (`composeSinks(consoleSink, jsonlFileSink(...))` at the daemon and `composeSinks(consoleSink, toastSink(...))` at the UI) failing to land for reasons other than priority : i.e. the composition shape proving wrong for one of them once a real implementation starts.

Any of those would justify revisiting. None is true today.
