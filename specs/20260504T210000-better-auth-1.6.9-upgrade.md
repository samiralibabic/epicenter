# Better Auth 1.5.6 to 1.6.9 upgrade

**Date**: 2026-05-04
**Status**: Draft
**Author**: AI-assisted (Claude)
**Branch**: TBD (single dependency-bump branch)

## One-Sentence Test

`bun.lock` resolves `better-auth`, `@better-auth/oauth-provider`, and `@better-auth/drizzle-adapter` to `1.6.9`, `drizzle-orm` resolves to `0.45.x`, the per-request `c.set('auth', ...)` baseURL flip in `apps/api/src/app.ts` is gone, the CLI schema config in `apps/api/better-auth.config.ts` lists the same plugins as the runtime, and `singleCredential` keeps the strict ambiguity rejection while its JSDoc cites the 1.6.6 fix instead of the unreleased changeset.

If the catalog still pins `^1.5.3`, the work is not done.
If `apps/api/src/app.ts` still has a middleware that rewrites `baseURL` per request, the work is not done.
If `apps/api/better-auth.config.ts` produces a different schema than `apps/api/src/auth/create-auth.ts`, the work is not done.
If `singleCredential` is gone, the work is not done (Better Auth 1.6.x still tolerates dual credentials by design).
If the spec adds two-factor, organization, OAuth dynamic client registration, or a desktop OAuth client, the work is not done (out of scope).

## Overview

Bump Better Auth from `1.5.6` to `1.6.9` and Drizzle from `0.44.7` to `0.45.x` as one dependency wave. Take the small simplifications the bump enables (one Hono middleware deletion, one CLI config bug fix) and leave every other piece of the auth surface intact. The bump is not a rewrite. The payoff is security hardening (1.6.6 SSRF/host classification, 1.6.7 cookie preservation, 1.6.4 PostgreSQL ID generation fix) and a slightly smaller surface, not a structural change.

This spec is also a **grounding document**. Section 1 walks through every Better Auth concept we use today, why we use it, and what the alternative would look like. If you read nothing else, read Section 1; the rest is the diff against it.

## Why this is its own spec

Per the cohesive-clean-breaks skill, a dependency bump is not on its own a spec-worthy change. What makes it spec-worthy here:

1. The bump is **coupled** to a Drizzle peer change (1.6.4 narrowed `drizzle-orm` peer to `^0.45.2`) and to a documented latent bug in our CLI config (`apps/api/better-auth.config.ts` does not list the runtime plugin set). Bundling them keeps the auth surface coherent across runtime and CLI.
2. The repo has accumulated several workarounds keyed to specific Better Auth quirks (`skipStateCookieCheck: true`, the manual `/.well-known/oauth-authorization-server` mount, the per-request `baseURL` flip for `wrangler dev`, the `singleCredential` rejection middleware). Each of those needs a yes/no answer against 1.6.x: keep, simplify, or delete. The spec is the place that answer lives.
3. The user explicitly asked for a clear-vision spec, with breaking changes allowed if a structural win exists. The honest answer (Section 4) is "no structural win exists; bump and take the small wins." That conclusion deserves to be written down so the next person doesn't re-do the research.

The repo precedent: see `20260503T230000-auth-unified-client-two-factories.md` and `20260504T030000-machine-auth-collapse-to-free-functions.md`. Each ships its own thesis-scoped spec even when the diff is small.

## 1. Grounding: our Better Auth surface today

### 1.1 What we use Better Auth for

Better Auth handles five concerns for Epicenter:

| Concern | Mechanism | Where |
|---|---|---|
| Email/password sign-in and Google OAuth | `betterAuth({ emailAndPassword, socialProviders.google })` | `apps/api/src/auth/create-auth.ts:67-72` |
| Cross-origin session cookies for browser apps | `advanced.crossSubDomainCookies` + `defaultCookieAttributes: { sameSite: 'none', secure: true }` | `create-auth.ts:109-118` |
| Bearer tokens for headless clients (CLI, daemons, browser extensions) | `bearer()` plugin | `create-auth.ts:211` |
| Device authorization (RFC 8628) for the CLI | `deviceAuthorization()` plugin | `create-auth.ts:213-218` |
| OAuth 2.1 authorization server (PKCE, RFC 8252 redirect URIs) | `oauthProvider()` plugin | `create-auth.ts:219-228` |

Two more cross-cutting plugins:

| Plugin | Purpose |
|---|---|
| `jwt()` | Issues JWTs the worker can verify without a session lookup. Currently mounted but no consumer actively reads them; reserved for downstream services. |
| `customSession(...)` | Enriches `/auth/get-session` responses with the per-user encryption keyring. |

### 1.3 The five workaround policies and why each exists

These are the places our config diverges from a vanilla `betterAuth(...)` call. Each is load-bearing; the upgrade does not change them unless noted.

| # | Policy | Lives in | Why it exists | After 1.6.9 |
|---|---|---|---|---|
| A | `account.skipStateCookieCheck: true` | `create-auth.ts:51-66` | Cross-origin sign-in POST: `opensidian.com → api.epicenter.so` cannot store the layer-2 state cookie because browsers drop third-party `Set-Cookie` from fetch responses (Chrome Privacy Sandbox, Safari ITP, Firefox ETP). Layer 1 (DB verification record) still validates the callback. | **Keep**. Deepwiki confirms 1.6.x still recommends this for cross-origin OAuth POSTs. |
| B | Manual `/.well-known/...` mounts | `app.ts:258-273` | `oauthProvider` warns it can't auto-mount discovery endpoints when `basePath` is `/auth` (not `/`). | **Keep**. Confirmed unchanged in 1.6.x. |
| C | Per-request `baseURL` flip for `wrangler dev` | `app.ts:157-165` | Wrangler dev serves the custom domain over plain HTTP (no TLS), so `Host: api.epicenter.so` arrives without `https://`; we rewrite to `http://localhost:8787`. | **Simplify**. 1.6.3 added object-form `baseURL` with `allowedHosts` + `protocol`, plus loopback host inference. We can move this into the static config and delete the middleware. |
| D | `singleCredential` middleware: reject cookie + bearer | `auth/single-credential.ts` | Better Auth's bearer plugin lets a bearer silently override a cookie. We want explicit rejection on ambiguity (no implicit precedence). | **Keep, update JSDoc**. The duplicate-cookie-header bug we cite (`fix-bearer-cookie-parse-mutate-serialize`) actually shipped in 1.6.6. The middleware still does work (rejection policy, WS subprotocol lift). |
| E | `customSession` for the encryption keyring | `create-auth.ts:238-251` | We ship the per-user encryption keyring inside `/auth/get-session` so clients can decrypt blobs without a separate key-fetch round trip. HKDF derivation is <0.1ms per key. | **Keep**. Deepwiki confirms `customSession` is still the only API for enriching session responses (no `databaseHooks` shortcut). |

### 1.4 What we deliberately do **not** use

| Surface | Why we don't use it | Worth revisiting? |
|---|---|---|
| `organization` plugin | DO names are user-scoped (Google Docs model), not org-scoped. See `app.ts:367-393`. | No, unless self-hosted enterprise emerges. |
| `twoFactor` plugin | No 2FA UI yet; risk of silently changing schema. | Defer (see 1.6.3 `verified` column note in §3). |
| `oidcProvider` (the deprecated one) | Replaced by `@better-auth/oauth-provider` per 1.6.0 deprecation. | We're already on the new one. |
| `passkey` / WebAuthn | Cross-origin browser support unclear with our `sameSite=none` cookies. | Defer. |
| `oauthProvider`'s dynamic client registration | `allowDynamicClientRegistration: false` (`create-auth.ts:223`); the CLI is the only client and its ID is hardcoded. | No. |

## 3. Design decisions

| Decision | Choice | Rationale |
|---|---|---|
| Bump `better-auth` to 1.6.9 | Yes | Security wins (1.6.6, 1.6.7) and forced Drizzle peer (1.6.4) make 1.5.6 the wrong place to sit. |
| Bump `@better-auth/oauth-provider` and `@better-auth/drizzle-adapter` to 1.6.9 | Yes (must move together) | They peer against the same `better-auth` version. Mixing 1.5.6 and 1.6.9 will fail peer resolution. |
| Bump `drizzle-orm` to `^0.45.2` | Yes (forced) | 1.6.4 narrows the peer. |
| Bump `drizzle-kit` | Match the runtime peer (`^0.31.x` line continues to support `drizzle-orm@0.45`) | Keep `drizzle-kit` and `drizzle-orm` minor lines compatible. Verify in §6. |
| Adopt `baseURL: { allowedHosts, protocol }` object form | Yes | 1.6.3 added it. Lets us delete the per-request middleware in `app.ts:157-165`. |
| Delete the per-request `c.set('auth', createAuth({ baseURL }))` middleware | Yes | Once `baseURL` is static (object form), `auth` becomes per-request only because of `db`. We can either (a) keep one middleware that only injects `db`-bound auth, or (b) memoize per `db` instance. (a) is simpler; pick that. |
| Move `auth` instantiation into a single per-request middleware that no longer reads `c.req.url` | Yes | The instantiation needs `db` (per-request), not `baseURL`. Cleaner separation. |
| Keep `singleCredential` middleware | Yes | 1.6.x still allows simultaneous cookie + bearer with bearer overriding silently (deepwiki confirmed). We want strict rejection. |
| Update `singleCredential.ts` JSDoc | Yes | Cite "1.6.6 fix-bearer-cookie-parse-mutate-serialize" instead of "2026 changeset." Pure documentation accuracy. |
| Keep `skipStateCookieCheck: true` | Yes | Cross-origin sign-in POST still needs it. Deepwiki confirmed no first-class replacement. |
| Keep `crossSubDomainCookies` + `defaultCookieAttributes` | Yes | Standard cross-origin auth-server pattern. |
| Keep `customSession` for encryption keyring | Yes | Only API for enriching `/auth/get-session`. |
| Keep manual `/.well-known/...` mounts | Yes | Still required when `basePath` is `/auth`. |
| Fix `apps/api/better-auth.config.ts` plugin omission | Yes (rides along) | Latent bug: schema generation will produce wrong tables when `oauthProvider`, `jwt`, `deviceAuthorization` schemas are involved. The CLI must see the same plugin set as the runtime. |
| Add `advanced.trustedProxyHeaders: true` | Defer | Open Question 1. Cloudflare sets `X-Forwarded-Proto`; turning this on may interact with the `baseURL` object form. Decide during implementation. |
| Tighten `AuthUser.email` to `'string | null'` for 1.6.8 nullable email types | Defer | Open Question 2. Google always returns email for our scope; pragmatic risk is low. |
| Two-factor, organization, passkey, dynamic client registration | Out of scope | Each is its own product decision, not a bump-time choice. |
| Run `bun run auth:generate` and diff the schema | Yes (verification gate) | 1.6.3 adds the `verified` column to `twoFactor`. Even though we don't enable 2FA, schema generation may emit it. Diff to a temp file first. |

## 4. Honest verdict

The bump is worth doing, but it is not a structural rewrite. Three categories of payoff:

**Real wins (do them):**
- Security: 1.6.6 host classification rewrite, 1.6.7 cookie preservation, 1.6.2 account-linking provider scoping. Each fixes a class of bug we could otherwise hit later.
- Required peer: 1.6.4 narrowed `drizzle-orm` to `^0.45.2`. We have to bump Drizzle anyway; doing it as part of the auth bump keeps the change cohesive.
- One middleware deletion: object-form `baseURL` lets us collapse the per-request `app.ts:157-165` flip.
- One latent bug fix: `apps/api/better-auth.config.ts` plugin list mismatch.

**Things 1.6.x does NOT let us delete:**
- `singleCredential` middleware (still needed; bearer/cookie ambiguity is a library default we override).
- `skipStateCookieCheck: true` (cross-origin OAuth still needs it).
- `customSession` enrichment for the encryption keyring (no `databaseHooks` shortcut).
- Manual `/.well-known/oauth-authorization-server/auth` and `/openid-configuration/auth` mounts (still required when `basePath` is `/auth`).
- Cross-subdomain cookie config.
- The CORS WebSocket-skip in `app.ts:108-122`.

**Future-only wins (do NOT do them now):**
- Multi-client-id idToken support (1.6.7) for per-platform Google sign-in.
- RFC 8252 loopback redirect matching (1.6.0/1.6.6) for desktop OAuth.
- `customTokenResponseFields` (1.6.3) for token endpoint customization.

**Net delta after the bump:**

```
   apps/api/src/app.ts                 ~ -10 lines (middleware C)
   apps/api/src/auth/single-credential.ts ~ -3 / +3 lines (JSDoc only)
   apps/api/src/auth/create-auth.ts    ~ +4 / -2 lines (baseURL object form)
   apps/api/src/auth/base-config.ts    ~ +30 lines (plugin list moves here)
   apps/api/better-auth.config.ts      ~ +2 lines (consume shared plugins)
   package.json (catalog)              ~ 5 version bumps
                                       ────────────
                                       net ~ -10 lines, schema unchanged
```

If you're hoping for a "big simplification" answer: there isn't one. The reason is that Better Auth 1.6.x did not change any of the integration points we are most coupled to (cross-origin OAuth, custom session enrichment, secondary storage with database fallback, OAuth provider discovery). The bump is the right move because of security and Drizzle, not because the new APIs unlock our code.

### 5.2 baseURL configuration

**Before** (in `app.ts:157-165`):

```ts
app.use('*', async (c, next) => {
    const origin = new URL(c.req.url).origin;
    const baseURL =
        origin === `http://${new URL(APPS.API.urls[0]).host}`
            ? `http://localhost:${APPS.API.port}`
            : origin;
    c.set('auth', createAuth({ db: c.var.db, env: c.env, baseURL }));
    await next();
});
```

**After** (in `base-config.ts`):

```ts
export const BASE_AUTH_CONFIG = {
    basePath: '/auth',
    baseURL: {
        // Better Auth resolves the base from the incoming request, validating
        // it against this allowlist. The protocol is inferred per host:
        // loopback hosts get http; everything else gets https unless overridden
        // by a trusted proxy header. See better-auth 1.6.3 release notes.
        allowedHosts: [
            ...new Set([
                ...Object.values(APPS).map((app) => new URL(app.urls[0]).host),
                `localhost:${APPS.API.port}`,
            ]),
        ],
    },
    emailAndPassword: { enabled: true },
    account: { /* ... */ },
} satisfies BetterAuthOptions;
```

And in `app.ts`:

```ts
app.use('*', async (c, next) => {
    c.set('auth', createAuth({ db: c.var.db, env: c.env }));
    await next();
});
```

### 5.3 CLI and runtime plugin alignment

**Before:**

```
BASE_AUTH_CONFIG (base-config.ts)
  └── basePath, emailAndPassword, account
                                              create-auth.ts (runtime)
                                                ├── ...BASE_AUTH_CONFIG
                                                ├── plugins:
                                                │     bearer, jwt,
                                                │     deviceAuthorization,
                                                │     oauthProvider,
                                                │     customSession
                                                └── socialProviders, secondaryStorage, ...

                                              better-auth.config.ts (CLI)
                                                ├── ...BASE_AUTH_CONFIG
                                                └── (no plugins)  ← bug
```

**After:**

```
BASE_AUTH_CONFIG (base-config.ts)
  ├── basePath, baseURL, emailAndPassword, account
  └── basePlugins: () => [bearer, jwt, deviceAuthorization, oauthProvider]
                                              create-auth.ts (runtime)
                                                ├── ...BASE_AUTH_CONFIG
                                                ├── plugins: [
                                                │     ...basePlugins(),
                                                │     customSessionPlugin,
                                                │   ]
                                                └── socialProviders, secondaryStorage, ...

                                              better-auth.config.ts (CLI)
                                                ├── ...BASE_AUTH_CONFIG
                                                └── plugins: basePlugins()
```

`basePlugins` is a function (not an array constant) because `oauthProvider` reads options at construction time and we want both call sites to get fresh instances.

`customSession` stays in `create-auth.ts` because it depends on the runtime `env` (via `deriveUserEncryptionKeys` and the env-coupled config). The CLI doesn't need it for schema generation; `customSession` does not add tables.

## 7. Edge cases

### 7.1 Wrangler dev with the object-form baseURL

1. Wrangler serves `http://api.epicenter.so` (HTTP, custom-domain Host header).
2. Request reaches Better Auth.
3. Better Auth checks `Host: api.epicenter.so` against `allowedHosts`. Hit.
4. Protocol inference: `api.epicenter.so` is not loopback. Without `trustedProxyHeaders`, Better Auth falls back to the request scheme.
5. **Risk**: if Better Auth defaults to `https` for non-loopback hosts and the request scheme is `http`, we get protocol mismatch.

Mitigation paths:
- Set `BETTER_AUTH_URL=http://localhost:8787` in Infisical's dev environment and let env override the host. This is the cleanest fix; production already has the same env not set, so the request URL drives.
- Or: add `localhost:8787` to `allowedHosts` and rely on the wrangler dev custom-domain config to route to localhost. (Wrangler's custom domain in dev is configurable.)
- Or: keep a one-liner middleware that flips `c.req.url` (not `baseURL`) to the localhost form before Better Auth sees it. Smaller than today's middleware.

The implementer should test all three paths during Wave 3.5 and pick whichever requires the least code.

### 7.2 Schema regeneration changing the `twoFactor` table

1.6.3 adds a `verified` column to the `twoFactor` table. We don't enable 2FA, but `auth:generate` may still emit the table.

If the diff includes `twoFactor.*` changes:
- Confirm the `twoFactor` table is not in the production schema (we never enabled the plugin).
- If the plugin's tables are unconditionally emitted, accept the diff but mark the table as unused (DB migration generates it; runtime never writes).

### 7.3 customSession + secondaryStorage interaction

Better Auth caches sessions in KV but deepwiki confirms `customSession` runs per request even when KV hits. So our `deriveUserEncryptionKeys` call runs every `/auth/get-session`. This is unchanged by the bump; the cost is <0.1ms per call (HKDF). If we ever care, the fix is downstream caching, not a Better Auth feature.

### 7.4 Bearer + cookie ambiguity in production

Our `singleCredential` middleware throws HTTP 400 on simultaneous credentials. If a buggy client sends both, that client breaks after the bump too. No behavior change.

### 7.5 Drizzle 0.44 → 0.45 type changes

Drizzle's minor versions occasionally change generic inference for `select()` and `insert()`. Likely a few one-line type fixes in `apps/api/src/app.ts` route handlers. Not a structural change.

## 8. Open questions

1. **Should we set `advanced.trustedProxyHeaders: true`?**
   - Cloudflare sets `X-Forwarded-Proto`. Turning this on lets Better Auth honor the upstream protocol, which would resolve the wrangler-dev edge case (§7.1).
   - **Recommendation**: defer. The object-form `baseURL` plus a `BETTER_AUTH_URL` env var in Infisical's dev environment is simpler. Add `trustedProxyHeaders` only if we hit a real production case where the protocol is wrong.

2. **Tighten `AuthUser.email` to allow `null` for 1.6.8 nullable provider profiles?**
   - Google always returns email when scope includes it. Apple, Facebook, etc. can omit.
   - **Recommendation**: defer. We only use Google. Add `mapProfileToUser` and a nullable email type when we add Apple or another provider that omits.

3. **Run schema regeneration before or after the runtime bump?**
   - Before: catches plugin-list-mismatch bugs early.
   - After: matches the bumped library's schema exactly.
   - **Recommendation**: after. Wave 6 explicitly runs it post-bump and pre-deploy.

4. **Should the Hono-side CORS allowlist change to mirror `BASE_AUTH_CONFIG.baseURL.allowedHosts`?**
   - Today they are computed independently in `app.ts:101-107` and `create-auth.ts:159-176`. Both derive from `APPS`, so they stay in sync, but the duplication is real.
   - **Recommendation**: defer. A future spec can extract a shared `getAllowedOrigins()` helper. Out of scope for the bump.

5. **Pin the catalog to an exact version (`1.6.9`) or accept caret (`^1.6.9`)?**
   - Caret accepts 1.6.10+ patch updates automatically.
   - **Recommendation**: caret. Patch versions in Better Auth have been small. Pinning would require us to manually bump for security patches.
